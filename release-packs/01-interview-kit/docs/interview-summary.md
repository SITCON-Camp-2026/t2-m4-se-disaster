# Main Agent 訪談彙整

> 這份文件可以由 Codex 先產生草稿，但你必須刪除、修改或補充。請保留人類確認痕跡。

## 核心結論

v1 優先服務「行動者」：現場志工、小隊長、運送協助者、新手志工、現場確認者，以及會把資訊交給行動者確認的人。

但 v1 不應幫他們做真實救災判斷。重點是讓他們看懂：

- 這不是正式任務。
- 哪些資訊不能直接行動。
- 缺少哪些人工確認。
- 未確認內容應轉成確認問題，而不是出發或派工指令。

## 共同需求

- 分清楚「原始資訊」「候選草稿」「確認問題」「正式任務」。
- 清楚標出 AI 推測、人工整理、原文可見內容的差別。
- 說明哪裡需要人工確認，不要只顯示籠統的「需要確認」。
- 讓行動者知道不能直接行動的原因，而不是只看到紅色警示或狀態詞。

## 主要衝突

- 回報者想快速留下不完整資訊；行動者需要足夠資訊才不會誤行動。
- 行動者想知道「現在可以做什麼」；安全邊界要求系統不能產生出發、派工或優先順序判斷。
- 資源提供者想提前準備；協調者與查核者擔心候選需求被誤傳成募集或任務。
- 查核者可能只確認某個欄位；其他人可能誤以為整筆資訊都已確認。

## 被推翻的 Phase 0 假設

原本以為只要標示不確定，使用者就能理解候選草稿不是任務。訪談顯示不夠。

候選草稿只要文字完整、語氣明確，就可能被行動者、新手志工或資源提供者誤讀成已確認。

## 最大風險

- AI 或介面把「可能」寫得太像「確定」。
- 相似資訊被誤認為互相佐證。
- 候選需求被誤讀成募集、派工或出發指令。
- 「已確認」沒有說清楚確認的是哪個欄位。

## v1 設計方向

- 以行動者視角呈現「不能行動原因」。
- 把未確認資訊轉成「待確認問題」。
- 對候選草稿使用保守語氣，避免像任務指令。
- 將資訊取得方式、查核狀態、是否能行動、需要確認欄位分開呈現。
- 保留原文入口，避免只看 AI 整理後文字。

## 不讓 AI 自動決定

- 是否已確認。
- 是否可以派工或出發。
- 是否把候選草稿轉成任務。
- 多筆相似資訊是否代表同一件事。
- 真實救災優先順序。

## 人類決策

- 採用：v1 優先服務行動者。
- 採用：新增 persona 對「候選草稿像任務」與「確認狀態太籠統」的警訊。
- 不採用：讓系統產生「可以出發 / 不可以出發」判斷。
- 不採用：現在直接進入 Release 02 flow design；Release 02 先擱置。

## 仍不確定

- 「低風險下一步」要怎麼寫，才不會變成變相派工。
- 行動者需要看到多少原始資訊才夠判斷風險。
- 局部確認要先用文字紀錄，還是需要新的資料格式。
