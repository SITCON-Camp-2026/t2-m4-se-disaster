# 需求取捨決策

> 這份文件可以由 Codex 先產生草稿，但最後必須由你刪除、修改或補充。請不要讓 AI 替你做最後決策。

## 我決定 v1 優先服務的使用者

請勾選一個主要對象，也可以補充原因。

- [ ] 回報者
- [ ] 資訊整理者
- [x] 行動者

原因：

人類決策：v1 優先服務行動者，也就是可能根據資訊採取下一步的人。這不代表系統要替他做真實救災判斷，而是要讓行動者更快看出「現在不能直接行動」「還缺哪些確認」「這只是候選草稿」。因為最危險的錯誤不是整理慢一點，而是有人把未確認資訊當成正式任務。

## 這個使用者最想完成什麼

- 快速分辨哪些資訊不能直接行動，若查看AI生成之草應提供警示，避免把候選草稿當成指令。
- 看懂一筆資訊缺少哪些關鍵確認，例如時間、地點、需求是否仍存在、誰能確認。
- 在資訊還沒確認時，知道應該停下來回報確認需求，而不是自行補完或出發。

## 這個使用者最怕什麼錯誤

- 把未確認資訊當成正式任務。
- 根據模糊或過期資訊跑錯方向、重複行動，或把錯誤需求轉傳出去。

## 我今天暫時不處理的需求

請至少寫一項。

- 暫時不讓系統產生「可以出發」「應該派誰」這類真實行動判斷。
- 暫時不使用 Release 02 flow design 來改寫訪談結論；Release 02 檔案已放回 repo，但先擱置。

## 我不採用的 AI 建議

請至少寫一項，並說明原因。

- AI 建議：
- 因為選擇行動者，就直接設計「可以出發 / 不可以出發」的任務判斷。
  - 不採用原因：
- 目前資料仍是未整理或待確認資訊，系統不應替使用者做真實行動判斷；v1 應優先讓行動者看出不能行動的原因與需要回報確認的地方。

## 必須保留人工確認的判斷

- 資訊是否已確認。
- 多筆相似資訊是否代表同一事件。
- 未確認資訊是否已交付確認人員。

## 今天明確不做什麼

- 不修改 `src/**` 或前端畫面。
- 不把候選草稿轉成正式任務。
- 不讓系統輸出真實派工、路線、優先順序或出發建議。
- 不使用真實地圖、地址、電話、人物或外部資料補完內容。

## 我仍不確定的地方

- 「低風險下一步」應該怎麼寫，才不會被誤讀成派工指令。
- 行動者畫面是否只顯示「不能行動原因」，還是也顯示整理者的候選摘要。
