# 其他角色取捨決策

> 這份文件補足目前 `decisions.md` 以行動者為主的視角。它不是要推翻行動者優先，而是先看回報者與資訊整理者需要什麼，之後再找一個三方都能接受的 v1 方案。

## 方案 A：優先服務回報者

- [x] 回報者
- [ ] 資訊整理者
- [ ] 行動者

原因：

如果先服務回報者，v1 的重點是讓人能快速留下資訊，而且不會被迫補完自己不知道的內容。系統應該清楚保留「這是本人確認、現場目擊，還是轉述」。

### 這個使用者最想完成什麼

- 快速送出原始資訊。
- 不知道欄位時也能先留下線索。
- 說清楚自己是當事人、目擊者，還是轉述者。
- 避免送出的內容被誤解成已確認事實。

### 這個使用者最怕什麼錯誤

- 表單太難，導致不想回報。
- 系統自動補完地點、數量、需求類型。
- 轉述內容被看成本人確認。
- 不確定描述被改寫成正式任務。

### 如果採用這個方向，v1 要做什麼

- 提供簡單回報表單。
- 允許留下不完整資訊。
- 回報者輸入時顯示可能相關的既有報告。
- 如果是同一主題，允許把新內容加在既有報告下方，避免重複建立新報告。
- 明確標示「本人確認 / 目擊 / 轉述 / 不確定」。
- 送出後預設進入待確認，不直接進入行動主畫面。

### 不做什麼

- 不強迫回報者填完所有欄位。
- 不自動補真實地點、座標、電話或人物。
- 不把回報內容改寫成任務。
- 不由系統自動判定兩筆報告一定是同一事件。

## 方案 B：優先服務資訊整理者

- [ ] 回報者
- [x] 資訊整理者
- [ ] 行動者

原因：

如果先服務資訊整理者，v1 的重點是把混亂資訊整理成可交接的判斷紀錄。整理者不是要讓所有資料都變成任務，而是要看出哪些資訊可信、哪些需要確認、哪些應暫時保留。

### 這個使用者最想完成什麼

- 快速看懂來源、原文、查核狀態。
- 分辨重複、衝突、缺漏資訊。
- 把未確認資訊整理成確認問題。
- 留下為什麼這樣判斷的紀錄。

### 這個使用者最怕什麼錯誤

- AI 補出原文沒有的內容。
- 查核狀態和任務狀態混在一起。
- 相似回報被誤合併。
- 下一位協作者分不清原文、推測、人工判斷。

### 如果採用這個方向，v1 要做什麼

- 顯示原文與候選整理的差異。
- 標出缺少哪些確認。
- 保留人工判斷紀錄。
- 提供「暫不採用」「需要確認」「可交給行動者查看」等分流。

### 不做什麼

- 不把所有資料強迫變成候選任務。
- 不讓 AI 決定資訊是真是假。
- 不讓 AI 自動合併相似回報。

## 三個角色一起看時的共同方向

目前比較合理的共同方案是：

1. 回報者負責快速留下原始資訊。
2. 資訊整理者負責把原始資訊分流與留下判斷紀錄。
3. 行動者只看適合行動者閱讀的資訊，並可切到未確認 / 高風險分頁查看警示資料。

這樣 v1 不需要假裝同時服務所有人，而是讓三個角色在同一條資訊鏈上各做一件事。

## 可能的最小 v1 流程

```text
回報者送出原始資訊
  → 系統顯示可能相關的既有報告
  → 回報者可選擇新增報告，或把內容加到既有報告下方
  → 預設標示為待確認
  → 資訊整理者查看原文與來源
  → 分流為：已人工確認 / 未確認高風險 / 疑似垃圾 / 暫不採用
  → 行動者主畫面只看較適合閱讀的資訊
  → 行動者可切到未確認分頁，但必須看到明顯警示
```

## 仍要由人決定

- 哪些報告可以出現在行動者主畫面。
- 哪些報告只能留在未確認 / 高風險分頁。
- 哪些欄位算是已人工確認。
- 相似回報是否代表同一件事。
- 高急迫但未確認的報告，要不要讓行動者看到。

## 不變的安全界線

- 不做真實救災判斷。
- 不自動派工。
- 不提供真實路線、座標、地圖連結或外部資料補完。
- 不把未確認資訊標成已確認。
- 不把 AI 推測寫成事實。
