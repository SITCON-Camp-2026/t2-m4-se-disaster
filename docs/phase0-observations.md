# Phase 0 Observations

## 我們做出了什麼？

- 首頁可以看到原始資訊列表與整理工作台。
- 整理工作台目前用前端記憶體建立、編輯、刪除與重設整理草稿，重新整理頁面後不保留。
- 草稿只表示小組嘗試整理，仍不是整理後資料，也不是已確認事實。

## 哪些資料讓 agent 很容易整理？

- M-010 有時間、數量、限制與下一次盤點時間，看起來比較容易整理成候選地點狀態。
- 但 M-010 仍是 `needs_review`，只能當候選整理，不能顯示成已確認。

## 哪些資料讓 agent 亂補或誤判？

- M-003 可能讓 agent 直接推成「需要水電任務」，但原文只說「比較需要水電」，還缺少需求範圍與確認者。
- M-004 可能讓 agent 把群組貼文當成雨鞋庫存事實，但來源是社群轉述，沒有現場盤點時間。

## 哪些資訊不能直接變成任務？

- M-001 缺少可確認地址、需求人、現場安全狀態，不能直接派十幾個人。
- M-005 截圖日期與來源不明，不能直接當道路封閉任務或公告。
- M-006 集合點資訊和淹水回報互相衝突，不能直接要求志工前往。
- M-012 來電者不在現場，親友位置與藥品需求都不確定，不能直接建立派工。

## 哪些資訊的操作者不是當事人？

- M-011 是現場志工代長者轉述，還要確認長者是否同意公開完整地址。
- M-012 是外地家屬來電，來電者無法確認親友目前位置。

## 我們剛剛偷偷假設了什麼？

- 把「看起來資訊較完整」當成「比較適合整理」，但這不等於已確認。
- 把「候選類型」當成有用分類，但目前分類仍是草稿欄位，不是正式資料格式。

## 如果要讓這個工具變可靠，下一步需要什麼？

- 需要明確定義哪些欄位由人類確認、哪些只是原文摘錄、哪些是 agent 推測。
- 需要確認角色與流程，避免任何人把草稿當成可派工資料。

## 我們需要更清楚資料格式的地方

- 需要區分原文證據、候選摘要、不能直接變成任務的原因、人工確認備註。
- 需要區分 `verificationStatus` 和草稿狀態；待確認不能被顯示成已確認。

## 我們需要人工確認的地方

- 來源是否可靠、時間是否仍有效、地點是否足夠明確。
- 操作者是否為當事人，以及是否同意公開或使用更完整的位置資訊。
- 任何看起來可以派人的內容，都需要先確認安全與現場需求。
